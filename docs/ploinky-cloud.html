Overview

Ploinky Cloud is a self-hosted serverless runtime platform that enables you to deploy Ploinky agents as Lambda-style functions. It provides an open-source alternative to commercial serverless platforms, giving you complete control over your infrastructure while maintaining simplicity.

Key Benefits

💰 Cost Efficient

Pay only for actual compute time. No idle costs, no vendor markup.

🔓 No Vendor Lock-in

Fully open-source. Deploy on any infrastructure, migrate anytime.

🚀 Auto-scaling

Automatically scales from zero to thousands of concurrent executions.

🛡️ Security First

Built-in authentication, container isolation, and secure secrets management.

🔧 Easy Integration

RESTful API, JavaScript SDK, and CLI for seamless integration.

📊 Monitoring

Built-in logging, metrics, and performance monitoring.

Architecture

Ploinky Cloud implements a lightweight container orchestration system optimized for serverless workloads.

┌─────────────────────────────────────────────────────────────────┐
│                         External Clients                         │
│  (Browser, p-cli, PloinkyClient, curl, 3rd-party integrations)  │
└────────────────────────────┬────────────────────────────────────┘
                             │ HTTP/HTTPS
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│                     Ploinky Cloud Server                         │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │                    Request Router                         │  │
│  │  - Path-based routing (/management, /auth, /agents/*)    │  │
│  │  - Domain/host routing for multi-tenancy                 │  │
│  └──────────────────┬───────────────────────────────────────┘  │
│                     │                                           │
│  ┌──────────────────▼───────────────────────────────────────┐  │
│  │                  Task Orchestrator                        │  │
│  │  - Task creation and scheduling                          │  │
│  │  - Deployment management                                 │  │
│  │  - Container lifecycle control                           │  │
│  └──────────────────┬───────────────────────────────────────┘  │
│                     │                                           │
│  ┌──────────────────▼───────────────────────────────────────┐  │
│  │              AgentCore Task Executor                      │  │
│  │  - Task queue management                                 │  │
│  │  - Result polling and retrieval                          │  │
│  └──────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘

Components

  • Request Router: Handles incoming HTTP requests and routes them to appropriate handlers
  • Task Orchestrator: Manages task lifecycle, deployment, and container operations
  • AgentCore: Runtime engine executing inside containers
  • Container Manager: Abstracts Docker/Podman operations
  • Guardian: Authentication and authorization layer

Quick Start

1. Start the Cloud Server

# Start Ploinky Cloud server (from ploinky directory) node cloud/index.js # Or with environment variables PORT=8000 NODE_ENV=production node cloud/index.js

2. Connect from CLI

# Connect to the server p-cli cloud connect localhost:8000 # Initialize and get API key p-cli cloud init # Login with API key p-cli cloud login YOUR_API_KEY

3. Deploy Your First Agent

# Add a repository p-cli cloud repo add demo https://github.com/PloinkyRepos/PloinkyDemo.git # List available agents p-cli cloud agent list # Deploy an agent p-cli cloud deploy localhost /hello hello-world # Test the deployment curl http://localhost:8000/hello

Connection & Authentication

cloud connect

cloud connect [url]
Connect to a Ploinky Cloud server
[url]
Server URL (default: localhost:8000)
Examples: cloud connect # Connect to localhost:8000 cloud connect api.example.com # Connect to remote server cloud connect 192.168.1.100:8080 # Connect with custom port
Note: Connection info is saved in .ploinky/cloud.json for future use

cloud init

cloud init
Initialize the server and generate an Admin API Key for first-time setup
Example: cloud init # Output: Admin API Key: ABCDEF123456 # Save this key securely!
Note: This command can only be run once when the server has no admin users. The generated API key is displayed only once - save it securely!

cloud login

cloud login <API_KEY>
Authenticate with the cloud server using an API Key
<API_KEY>
Admin API Key generated during initialization
Examples: cloud login ABCDEF123456 cloud login 7b9d8e6f5a4c3b2a1d0e9f8g7h6i5j4k
Note: The API key is stored locally for future requests. Use 'cloud logout' to remove it.

cloud logout

cloud logout
Logout from the cloud server and remove stored credentials
Example: cloud logout

cloud status

cloud status
Display current connection and authentication status
Example output: Server: http://localhost:8000 Status: Connected ✓ Authenticated: Yes User: admin Deployments: 3 active

cloud show

cloud show
Display current cloud URL and API Key
Example: cloud show

Host Management

cloud host add

cloud host add <hostname>
Register a new hostname or domain for routing
<hostname>
The hostname or domain to register (e.g., example.com, api.myapp.io)
Examples: cloud host add example.com cloud host add api.myapp.io cloud host add staging.internal.net
Note: The hostname must point to your Ploinky Cloud server's IP address in DNS

cloud host remove

cloud host remove <hostname>
Remove a registered hostname
<hostname>
The hostname to remove
Example: cloud host remove staging.internal.net
Note: This will also remove all deployments associated with this hostname

cloud host list

cloud host list
List all registered hostnames
Example output: Registered hosts: - localhost (3 deployments) - example.com (5 deployments) - api.myapp.io (2 deployments)

Repository Management

cloud repo add

cloud repo add <name> <url>
Add a Git repository containing Ploinky agents to the cloud
<name>
A unique name for the repository
<url>
Git URL (HTTPS or SSH) of the repository
Examples: cloud repo add MyAgents https://github.com/user/my-agents.git cloud repo add PrivateAgents git@github.com:company/private-agents.git cloud repo add Demo https://github.com/PloinkyRepos/PloinkyDemo.git
Note: The repository will be cloned to the server. For private repos, ensure the server has appropriate SSH keys configured.

cloud repo remove

cloud repo remove <name>
Remove a repository from the cloud
<name>
The repository name to remove
Example: cloud repo remove MyAgents
Warning: This will stop all agents from this repository and remove the repository files

cloud repo list

cloud repo list
List all repositories registered in the cloud
Example output: Cloud repositories: - Demo: https://github.com/PloinkyRepos/PloinkyDemo.git (5 agents) - MyAgents: https://github.com/user/my-agents.git (3 agents) - PrivateAgents: git@github.com:company/private-agents.git (10 agents)

Deployment Management

cloud deploy

cloud deploy <host> <path> <agent>
Deploy an agent to a specific URL path on a host
<host>
Target hostname (must be registered with 'cloud host add')
<path>
URL path where the agent will be accessible (e.g., /api, /webhook)
<agent>
Agent name to deploy
Examples: cloud deploy localhost /hello hello-world cloud deploy example.com /api api-gateway cloud deploy api.myapp.io /v1/auth auth-service
Note: After deployment, the agent will be accessible at http://host/path. The agent's container will be started automatically if not already running.

cloud undeploy

cloud undeploy <host> <path>
Remove a deployment from a specific path
<host>
Hostname where the agent is deployed
<path>
URL path of the deployment
Examples: cloud undeploy localhost /hello cloud undeploy example.com /api
Note: Route management is now handled by start/status. Use 'stop' to stop local containers.

cloud deployments

cloud deployments
List all active deployments across all hosts
Example output: Active deployments: localhost: /hello → hello-world (running) /echo → echo-server (running) example.com: /api → api-gateway (running) /webhook → webhook-handler (stopped) api.myapp.io: /v1/auth → auth-service (running) /v1/users → user-service (running)

Server Management

cloud destroy

cloud destroy <agents|server-agents>
Stop and remove agent containers
agents
Remove local containers (.ploinky/agents)
server-agents
Remove containers on the connected cloud server
Examples: cloud destroy agents # Remove local containers cloud destroy server-agents # Remove server containers
⚠️ Warning: This is a destructive operation. All containers will be stopped and removed.

cloud logs

cloud logs [lines|list|download <date>]
View and manage server logs
[lines]
Number of recent log lines to display (default: 100)
list
List available log files
download <date>
Download logs for a specific date (YYYY-MM-DD format)
Examples: cloud logs # Show last 100 lines cloud logs 500 # Show last 500 lines cloud logs list # List available log files cloud logs download 2025-09-01 # Download logs for Sep 1, 2025

cloud settings

cloud settings <show|set>
View or modify server settings
Examples: cloud settings show # Display all settings cloud settings set logLevel debug # Set log level to debug cloud settings set metricsRetention 365 # Keep metrics for 365 days cloud settings set maxConcurrent 100 # Max concurrent tasks
Available settings:
  • logLevel: error, warn, info, debug
  • metricsRetention: Days to keep metrics (1-365)
  • maxConcurrent: Maximum concurrent tasks (1-1000)
  • enableAuth: Enable/disable authentication (true/false)

API Reference

REST API Endpoints

Management API

GET /management/status # Server status GET /management/config # Configuration POST /management/config # Update configuration GET /management/deployments # List deployments POST /management/deployments # Create deployment DELETE /management/deployments/{id} # Remove deployment GET /management/agents # List agents GET /management/agents/{name} # Agent details POST /management/agents/{name}/start # Start agent POST /management/agents/{name}/stop # Stop agent

Authentication

POST /auth/login # Login with API key POST /auth/logout # Logout GET /auth/status # Authentication status

Agent Execution

POST /{host}/{path} # Execute agent function GET /{host}/{path} # Get agent information (if supported)

JavaScript SDK

// Initialize client const { PloinkyClient } = require('@ploinky/client'); const client = new PloinkyClient('http://localhost:8000'); // Authenticate await client.login('YOUR_API_KEY'); // Execute agent function const result = await client.call('/hello', 'greet', { name: 'World' }); console.log(result); // "Hello, World!" // Deploy an agent await client.deploy('localhost', '/api', 'my-agent'); // List deployments const deployments = await client.getDeployments(); console.log(deployments);

Authentication

All management endpoints require authentication via API key. Include the key in the Authorization header:

Authorization: Bearer YOUR_API_KEY

Best Practices

🔐 Security

  • Always use HTTPS in production
  • Keep API keys secure and rotate regularly
  • Use environment variables for secrets
  • Enable authentication for all deployments

🚀 Performance

  • Keep agents stateless for better scaling
  • Use appropriate resource limits
  • Monitor logs and metrics regularly
  • Cache frequently accessed data

📦 Deployment

  • Test agents locally before deploying
  • Use meaningful paths for deployments
  • Document your API endpoints
  • Implement health checks

🔧 Maintenance

  • Regular backups of configuration
  • Monitor disk space for logs
  • Update agents and dependencies
  • Plan for zero-downtime updates

Troubleshooting

Common Issues

Connection refused

If you can't connect to the server:

  • Verify the server is running: ps aux | grep "node cloud"
  • Check the port is correct: default is 8000
  • Ensure firewall allows the port: sudo ufw allow 8000
  • Check server logs: cloud logs 100

Authentication failed

If authentication fails:

  • Verify API key is correct: cloud show
  • Check if authentication is enabled: cloud settings show
  • Try logging out and back in: cloud logout && cloud login API_KEY

Agent not starting

If an agent won't start:

  • Check agent logs: cloud logs | grep agent-name
  • Verify container runtime: docker ps or podman ps
  • Check resource limits: Memory or CPU constraints
  • Verify repository is cloned: cloud repo list
