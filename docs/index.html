Ploinky: Secure & Simple Agent Environments

The "Why": Running Code in the Age of AI

In modern development, we increasingly use code from sources that are difficult to fully vet. This includes complex open-source libraries, scripts from the internet, and, most significantly, code generated by AI.

Running this code directly on your machine poses a security risk. A vulnerable or malicious script could potentially access your entire file system, read sensitive data, or cause damage beyond its intended scope.

Ploinky solves this problem. It provides a lightweight, secure sandbox for running code, ensuring that its access is strictly limited to the current project directory.

The "How": Simple, Powerful Isolation

Ploinky's core concept is simple: any script or tool can be defined as an Agent. Each agent runs inside its own isolated Linux container.

The magic is that Ploinky automatically maps your current working directory (e.g., /home/user/my-project) to the exact same path inside the container. The agent sees your project files and nothing else. It cannot see /home/user/ or any other part of your host file system.

This approach provides robust security with a seamless user experience. You can interact with tools inside the container as if they were running locally, but with the peace of mind that they are safely sandboxed.

In-Depth Guide: How Ploinky Works

Core Concept

The Ploinky environment enables the transformation of any bash script into a separate, isolated agent. These agents run inside a Linux container but share the host's working directory, which is mapped as a volume with the exact same path.

When you first run ploinky in a directory, it creates a hidden .ploinky folder. This folder, known as the "repos folder," contains subfolders for all your configured agent repositories.

Agent Management & Naming

An agent is a specific Docker container instance. The first time you run an agent, a container is created for it. On this initial run, the agent's install command is executed, followed by its run command.

Agent names must be unique. If you have agents with identical names in different repositories, you must use a prefixed name: RepoName:AgentName. If you use a short, non-prefixed name that is ambiguous (exists in multiple repos), Ploinky will throw an error and list the full, prefixed names for you to use.

Command Reference

All commands can be run directly (e.g., ploinky add repo ...) or within the interactive shell.

Repository Management

  • add repo <RepoName> <GitHubURL>
    Clones a repository using git clone. This repo should contain pre-configured agents, each in its own subfolder with a src folder and a manifest.json file.

Agent Configuration

  • new agent <RepoName> <AgentName> [ContainerImage]
    Creates the configuration for a new agent within the specified repository. If [ContainerImage] is omitted, it defaults to a versatile Debian-based image.
  • set install <AgentName> "[command]"
    Configures the command to be executed once when the agent's container is first created. By convention, this should install software globally within the container.
  • set update <AgentName> "[command]"
    Configures the command for updating the agent's internal software.
  • set run <AgentName> "[command]"
    Configures the primary command to be executed. This command runs within the user's current directory, which is mapped inside the container.

Environment Variables & Secrets

Securely manage secrets for your agents. These are stored in .ploinky/.secrets, which should be added to your .gitignore.

  • add env <VarName> <VarValue>
    Creates an environment variable in the .secrets file.
  • enable env <AgentName> <VarName>
    Makes an environment variable from the .secrets file available to a specific agent's working environment.

Running Agents

  • run agent <AgentName> [...args]
    Runs the agent's primary command with any subsequent arguments passed to it. The command is run interactively inside the container.
  • run bash <AgentName>
    Starts a new interactive bash session inside the agent's container. This is useful for debugging or manual operations.
  • run update <AgentName>
    Executes the agent's configured update command.

Listing and Discovery

  • list agents
    Lists all available agents, grouped by the repository they belong to. This is useful for seeing what tools are at your disposal.
  • list repos
    Lists all the repositories that have been added to your Ploinky environment.

Help & Development

  • help
    Displays the list of commands and their explanations.
  • test <test_name>CLI Only
    Runs a specific integration test from the /tests directory.
