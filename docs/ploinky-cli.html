Overview

The Ploinky CLI provides a comprehensive interface for managing containerized agents, both locally and in the cloud. It simplifies container orchestration while providing powerful features for developers.

Installation

Basic Installation

# Clone the repository git clone https://github.com/PlonkyRepos/ploinky.git cd ploinky # Add to PATH export PATH="$PATH:$(pwd)/bin" # Or create system-wide links (optional) sudo ln -s $(pwd)/bin/ploinky /usr/local/bin/ploinky sudo ln -s $(pwd)/bin/p-cli /usr/local/bin/p-cli

First Run

# Start the interactive CLI p-cli # Or run direct commands p-cli add repo basic p-cli list agents

Local Development Commands

üîß Repository Management

add repo

add repo <name> [url]
Add an agent repository to your local environment
<name>
Repository name (cloud/vibe/security/extra for predefined, or custom name)
[url]
Git URL for custom repositories (optional for predefined repos)
Examples: add repo cloud # Add predefined cloud repository add repo myrepo https://github.com/user/repo.git # Add custom repo
Note: Predefined repos include: cloud (AWS/Azure/GCP), vibe (social), security (auth/crypto), extra (utilities)

üöÄ Agent Management

new agent

new agent <repo> <name> [image]
Create a new agent in a repository
<repo>
Repository name where agent will be created
<name>
Agent name (must be unique)
[image]
Container image (default: node:18-alpine)
Examples: new agent cloud MyAPI # Node.js agent new agent cloud PyBot python:3.11 # Python agent new agent vibe WebApp nginx:alpine # Nginx agent
Note: Creates manifest.json and basic structure in .ploinky/repos/<repo>/<name>/

update agent

update agent <name>
Modify container, install, update, cli, agent, about fields interactively
Example: update agent MyAPI

refresh agent

refresh agent <name>
If agent command exists, stop/remove/restart container; otherwise stop/remove only
Example: refresh agent MyAPI

‚ñ∂Ô∏è Running Agents

cli

cli <name> [args...]
Run manifest "cli" command interactively (attached TTY). Prompt returns when the command exits.
<name>
Agent name
[args...]
Arguments appended to the cli command
Example: cli MyAPI --help
Note: Always attaches to container TTY. If the cli is a REPL (e.g., node/python), stay attached until you exit.

shell

shell <name>
Open interactive POSIX sh (attached TTY) in the agent container
Example: shell MyAPI # Opens shell with current workspace mounted
Note: Always attaches to container TTY and waits until you exit the shell.

webconsole

webconsole <password> [command...]
Start WebTTY (Console/Chat) attached to a local command. Defaults to /bin/bash if no command is provided.
<password>
Password required for UI access
[command...]
Local command to run inside the PTY (e.g., p-cli shell MyAgent)
Examples: webconsole secret webconsole secret p-cli shell MyAPI
Note: UI served on localhost:WEBTTY_PORT (default 8089). Set title via WEBTTY_TITLE. Supports resize (node-pty). Auth via POST /auth.

start

start [staticAgent] [port]
Start enabled agents and the local Router. On first run, provide a static agent and port.
[staticAgent]
Agent whose static files are served and whose manifest directives (repos/enable) are applied
[port]
Router port (default 8088)
Examples: start MyWeb 8088 start
Note: Writes routes to .ploinky/routing.json and keeps the Router attached.

üîë Variables & Environment

set

set <VAR> <$OTHER|value>
Set a workspace variable to a literal value or alias another variable.
set API_KEY sk-123456 set PROD_KEY $API_KEY
Note: Variables are stored in .ploinky/.secrets. set without arguments lists variable names.

echo

echo <VAR|$VAR>
Print the resolved value of a variable (aliases resolved).

expose

expose <ENV_NAME> <$VAR|value> [agent]
Expose a variable or literal to the agent's environment; injected on container create.
expose API_KEY $PROD_KEY MyAPI expose MODE prod MyAPI

cloud connect

cloud connect [url]
Connect to a Ploinky Cloud server
[url]
Server URL (default: localhost:8000)
Examples: cloud connect # Connect to localhost:8000 cloud connect api.example.com # Connect to remote server cloud connect 192.168.1.100:8080 # Connect with custom port
Note: Connection info saved in .ploinky/cloud.json

cloud login

cloud login <API_KEY>
Login to connected cloud server using API Key
<API_KEY>
Admin API Key (generated with cloud init)
Examples: cloud login ABCDEF123456 cloud login 7b9d... (hex key)
Note: Use cloud init first to generate an API Key

cloud init

cloud init
Initialize server and generate Admin API Key
Note: Stores URL and API Key in ~/.plionky/remotes.json

cloud logout

cloud logout
Logout from cloud server

cloud status

cloud status
Show connection and authentication status
Note: Shows server URL, login status, and deployment info

Client Operations

üîå Agent Interaction

client task

client task <agent>
Interactive: asks for command type and parameters (multiline until "end"), then POSTs JSON to /apis/<agent> on RoutingServer
<agent>
Agent name
Example: client task MyAPI

client methods

client methods <agent>
List available methods for an agent (if implemented by agent)
Examples: client methods MyAPI client methods DataProcessor

client status

client status <agent>
Get runtime status of an agent (if implemented by agent)
Example: client status MyAPI
Note: Shows state, uptime, resource usage, and recent activity

client list

client list
List all available agents

client task-status

client task-status <agent> <task-id>
Check the status of a submitted task
<agent>
Agent name
<task-id>
Task ID returned when task was submitted
Example: client task-status MyAPI task-123

Best Practices

üèÅ Getting Started

  • Start with predefined repositories (basic, cloud, vibe)
  • Use list agents to explore available agents
  • Test agents with cli before deploying

üîê Security

  • Always use add env for sensitive data
  • Enable variables per-agent with enable env
  • Use strong passwords for WebTTY access

üöÄ Development

  • Use shell for interactive debugging
  • Keep agents stateless when possible
  • Version control your manifest.json files

‚òÅÔ∏è Cloud Deployment

  • Test locally before cloud deployment
  • Use meaningful hostnames and paths
  • Monitor with cloud logs regularly
